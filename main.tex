\section{Introduction}
Science fiction has always been reliable at predicting what we want before we know how to build it. The communicator from Star Trek became the first commercial flip phone. The tablet computer from 2001: A Space Odyssey became the iPad. These weren’t coincidences. They were the same human desire, expressed first in story, then in silicon.

Of all the technologies science fiction imagined, the Holodeck stuck around the longest without a real answer. A user walked in, described a world, and the system built one around them. No menus, no asset browsers, no friction between imagination and experience. It felt further away than flying cars because the problem it was solving was harder to name. The hardware was the easier problem to solve. Understanding what a person meant, building something coherent from it, and delivering it before the moment of anticipation turned into waiting was the actual problem.
That sequence of steps is deceptively demanding. A user speaks a sentence. The system interprets intent, selects a layout, furnishes it with appropriate objects, themes it correctly, and verifies that every surface is safe to walk on. In VR specifically, all of that has to work before the user takes a single step. The system earns the experience before it offers it.

Recent work has made real progress on pieces of this. ProcTHOR demonstrates that large-scale valid indoor environments can be generated procedurally with measurable generalization benefits for embodied AI~\cite{Deitke_2022_NeurIPS}. Text2Room shows that a text prompt can seed a spatially consistent 3D mesh by lifting 2D generative outputs into a 3D representation~\cite{hoellein2023text2room}. Work directly named Holodeck has explored language-guided 3D environment generation by selecting and placing assets to satisfy language-specified constraints~\cite{Yang_2024_CVPR}. Each of these advances a different part of the problem. None of them put a user inside a headset and asked the system to perform in real time.

This paper describes a VR-first system in which a user enters a stable origin room, submits a natural language prompt by voice or text, and steps through a portal into a compiled, walkable destination world. The client application is implemented in Unity using C\# and targets the Meta Quest platform via the OpenXR runtime. Prompts are converted into a validated intermediate representation called WorldSpec, which a deterministic compiler transforms into a runtime-executable world bundle. To reduce perceived latency and prevent disorienting world-formation effects, the destination world appears first as a walkable greybox and upgrades visually in streaming phases as assets load. The system is evaluated on time to first walkable world, compilation success rate, constraint satisfaction, and runtime performance under Quest-native conditions.
\section{Background}
\subsection{Large Language Models and Structured Output}
Large language models (LLMs) are neural networks trained on large corpora of text to model and generate natural language. When prompted with appropriate constraints, these models produce outputs that conform to structured formats, including JSON schemas, enabling their integration as planning components in software pipelines. Prior work has demonstrated this planning pattern by mapping natural language into structured layout representations suitable for downstream scene construction~\cite{Feng_2023_NeurIPS_LayoutGPT}. Schema-locking, which constrains model output to a predefined JSON schema, reduces the likelihood of hallucinated or invalid field values and allows downstream components to parse and validate model responses deterministically. This property is central to the system described in this paper, which uses an LLM to convert natural language prompts into a validated WorldSpec representation rather than generating 3D content directly. In the current system, malformed or out-of-registry model output fails explicitly at validation boundaries rather than dropping into an unconstrained heuristic fallback path.

\subsection{Virtual Reality and OpenXR}
Virtual reality systems present stereoscopic rendered environments through a head-mounted display, tracking head and controller motion to update the view in real time. The OpenXR standard defines a cross-platform API for extended reality applications, allowing a single codebase to target multiple headset platforms~\cite{Khronos_OpenXR_Spec}. The Meta Quest line of headsets runs OpenXR natively and imposes strict performance constraints: frame time budgets, draw call limits, and texture memory ceilings must be respected to maintain visual stability and user comfort. Teleport locomotion, a standard interaction pattern in VR, allows a user to select a destination on a valid floor surface and reposition instantly, sidestepping continuous motion that can cause discomfort. These constraints shaped the design of this system at every level, from asset budget enforcement to the decision to prioritize greybox compilation before visual quality.

\subsection{Asset-Based Scene Composition}
Asset-based scene composition constructs environments by selecting and placing pre-authored 3D objects from a controlled library, rather than generating geometry procedurally from noise functions or lifting it from 2D outputs. This approach trades generative range for reliability: the resulting environments contain assets with known geometry, collision data, and performance characteristics that a runtime can depend on. Large-scale procedural environment generation systems such as ProcTHOR illustrate this trade-off by composing valid scenes from constrained asset sets and layout rules~\cite{Deitke_2022_NeurIPS}. Procedural placement systems arrange these assets according to spatial constraints and layout rules while preserving physical validity. This system adopts asset-based composition as its primary generation strategy, supplemented by procedurally generated greybox geometry for structural layout. Assets enter the planner-safe inventory only after passing a manual review pipeline that validates semantic quality and assigns exclusion flags where appropriate.

\subsection{Compiler Pipelines}
Compiler pipelines transform high-level input into validated intermediate representations before producing executable outputs. Classic compiler design formalizes this staged structure as lexical and syntactic analysis, semantic checks, intermediate representation generation, optimization, and target code emission~\cite{Aho_2006_Compilers}. The same architectural pattern is useful for prompt-to-world systems: natural language is parsed into a constrained WorldSpec, validated for schema and safety constraints, and then compiled into runtime-ready artifacts. Treating world generation as a compiler pipeline makes failures easily identifiable at stage boundaries, supports explicit fail-fast validation, and reduces runtime surprises in VR.

\section{Related Work}
Research in prompt-driven 3D environment generation has split along one foundational question: should a system generate novel geometry from a prompt, or assemble environments from a curated library of pre-authored assets? That choice has consequences for reliability, performance, and what guarantees a downstream runtime can actually make.
Text2Room takes the generative geometry approach. Hollein et al. showed that a text prompt can produce a spatially consistent, textured 3D mesh by lifting outputs from 2D image generation models into a 3D representation~\cite{hoellein2023text2room}. The strength of this approach is coverage. Because geometry is synthesized rather than retrieved, the system can respond to arbitrary prompts without being limited by an asset catalog. In a live VR context, however, synthesized geometry requires post-generation analysis to determine walkability, draw call cost, and memory footprint before a runtime can safely load it. That additional step adds latency and introduces failure modes at the validation stage that are difficult to recover from within an interactive session. 3D-SceneDreamer addresses one dimension of this problem by representing global scene structure in a stable tri-plane NeRF and refining content in stages~\cite{Zhang_2024_CVPR_SceneDreamer}. The staged approach improves cross-view consistency and reduces cumulative geometric error, and that architectural pattern, stabilizing global structure first then refining locally, maps onto the greybox-first compilation strategy this system uses. The difference is that geometry synthesis still produces outputs without known collision or performance properties, whereas asset-based compilation carries those properties from the start.

The asset-based approaches accept a narrower generative range in exchange for stronger guarantees. Deitke et al. and Yang et al. both work within pre-authored asset libraries, using structured representations and constraint satisfaction to produce environments whose physical properties are known before the world loads~\cite{Deitke_2022_NeurIPS, Yang_2024_CVPR}. Deitke et al. focus on procedural layout generation at scale, producing large collections of varied indoor environments for embodied AI training~\cite{Deitke_2022_NeurIPS}. Yang et al. connect natural language to asset selection and spatial placement directly, using a language model to convert prompts into structured scene descriptions that get compiled into interactive environments~\cite{Yang_2024_CVPR}. Both systems treat generation as a pipeline with explicit validation stages, and both produce environments reliable enough to train agents in. Both systems assume the asset library contains what the prompt requests. Asset coverage is an implicit prerequisite, not a problem the pipeline resolves. Feng et al. extend this direction by showing that large language models can serve as visual planners, converting natural language prompts into stylesheet-like structured layouts through in-context demonstrations~\cite{Feng_2023_NeurIPS_LayoutGPT}. Their results support a clear separation of concerns: planning in structured symbolic space first, then compilation afterward. That separation directly motivates the WorldSpec stage in this system, where the language model produces a schema-locked intermediate representation rather than geometry, and a deterministic compiler handles the rest.

A related body of work approaches structured scene representation through scene graphs, where objects are nodes and spatial relationships are edges. GraphDreamer uses scene graph conditioning to generate compositional 3D scenes, introducing object-wise disentanglement and relationship-aware constraints to reduce interpenetration and improve spatial coherence~\cite{Gao_2024_CVPR_GraphDreamer}. CommonScenes takes a similar position, combining a layout branch and a shape generation branch conditioned on scene-object and object-object relationships, and reports improved semantic consistency compared with retrieval-heavy alternatives~\cite{Zhai_2023_NeurIPS_CommonScenes}. More recent work on functional 3D scene graphs extends this idea beyond plain spatial relationships by representing interactive elements and functional links between them; OpenFunGraph is one example of this direction in real-world indoor spaces~\cite{Zhang_2025_CVPR_OpenFunGraph}. Together, these works provide evidence that explicit relational structure, represented before rendering rather than inferred from it, produces more controllable and reproducible outputs. For the purposes of this system, the scene graph functions as an analogue to WorldSpec: a machine-checkable intermediate representation that the compiler can validate against spatial and budget constraints before any geometry is loaded.

All three systems treat generation as an offline or training-time problem, where compilation time and frame stability fall outside the design requirements. A user standing in a headset waiting for a portal to open operates under a different set of conditions. The system described in this paper sits within the asset-based tradition of Deitke et al. and Yang et al., adopting structured intermediate representations and constraint-driven compilation for the same reliability reasons, but applies them to a live interactive setting where those guarantees have to hold in real time~\cite{Deitke_2022_NeurIPS, Yang_2024_CVPR}. A greybox-first compilation phase ensures a walkable world exists before visual assets begin streaming. A substitution system replaces missing assets with the closest tagged alternatives, keeping the pipeline moving regardless of asset coverage gaps. The contribution is a composition of these properties into a pipeline that maintains reliability guarantees under the constraints of a standalone VR headset while keeping semantic authorship, grounding, and deterministic validation separated into explicit stages.

\section{Methodology}
\subsection{Client (VR application)}
The client consists of the Unity-based VR runtime, the Creation Room, the in-world assistant panel, the portal transition system, and the destination-world executor. Together, these components handle prompt entry, user feedback, comfort-preserving movement, and the final handoff into the compiled world. The client enforces the runtime interaction contract by reading backend-produced manifest and phase artifacts, preparing the portal only after a valid destination is available, and relocating the XR origin to a validated safe spawn point on entry. In the current implementation, the Creation Room uses a faceted visual chamber with a separate curved collision shell, so both standing and teleport logic clamp the player against the actual chamber boundary rather than an earlier square-room approximation. It also consumes backend scene context directly: room archetype, decor plan, exact asset realizations, and validated transforms are all passed in rather than improvised on the headset. At runtime, the client applies comfort and presentation logic such as teleport-surface enforcement, collision-aware destination execution, player-facing panel recentering, destination-world polish, and return-home transitions while keeping headset-side behavior predictable.

\subsection{Backend (planning, compilation, and validation)}
The backend consists of the planner, WorldSpec validator, pack and stylekit registries, substitution layer, safe-spawn search, decor planner, runtime realization registry, and the phase0 compiler that writes the runtime contract. Together, these components convert a prompt into a validated manifest and phase artifact that the Unity client can execute predictably.

\subsubsection{Backend Pipeline}
Planning now occurs as a staged semantic pipeline. First, the language model produces a design brief that captures concept, palette strategy, material hierarchy, lighting layers, and other high-level authorship cues. Second, the model converts the prompt and design brief into a structured scene program built around semantic slots, groups, typed relations, density intent, walkway intent, and explicit negative constraints rather than only a flat list of runtime roles~\cite{Feng_2023_NeurIPS_LayoutGPT, Yang_2024_CVPR}. Third, a small deterministic completion pass fills a few high-value functional gaps, such as bedside support for sleeping spaces or work surfaces for work-oriented rooms, without turning the backend into a large hand-authored room recipe system. DiffuScene treats scene completion as a first-class capability in indoor scene generation, which makes that design choice easier to justify as a real scene-generation concern rather than an ad hoc patch~\cite{Tang_2024_CVPR_DiffuScene}. Fourth, asset selection operates over the reviewed shortlist as its own model-mediated decision, returning slot-based asset choices, slot-first group assignments, and structured fallback or rejection information. Only after that selection step are semantic slots grounded down to runtime roles and subtypes for the deterministic solver and compiler. If any stage fails or returns invalid identifiers, the request fails explicitly instead of falling to a heuristic fallback path.

Compilation turns that validated WorldSpec into something Unity can actually use. The compiler resolves missing assets through constrained substitution, builds the destination shell, runs deterministic layout solving over the grounded core furniture set, computes a safe spawn, and writes the manifest and phase0 artifact that the client reads at runtime. The runtime contract carries more than raw transforms. It also includes explicit grounded roles, scene context, and a decor plan so the room can be realized as a coherent space rather than as a loose set of objects. That scene context is now richer than earlier versions: alongside archetype and decor guidance, the backend carries concept labels, creative tags, mood tags, style descriptors, and approved material context forward so later material and presentation choices do not have to reconstruct intent from the original prompt. Validation runs through the whole process: schema checks catch malformed WorldSpecs, semantic checks catch missing functional coverage, grounding checks catch invalid asset assignments, post-placement checks catch overlap and entry problems, and budget checks help preserve Quest-compatible performance before the portal is allowed to open.

This split matters because a single ``generate everything'' step makes it hard to tell what actually went wrong. A bad result can come from prompt interpretation, asset selection, placement, or runtime realization. Breaking the work into stages makes those failures visible and makes debugging more concrete.

\subsubsection{Semantic Planning}
The semantic planning stage is built around a structured scene program rather than a free-form description. In practice, the planner first asks the language model for a design brief, then asks for a scene program built around semantic slots, slot-level or concept-level relations, groups, density, symmetry, walkway intent, scene features, and broad style cues~\cite{Feng_2023_NeurIPS_LayoutGPT, Gao_2024_CVPR_GraphDreamer, Zhai_2023_NeurIPS_CommonScenes}. The move toward semantic slots and support hierarchy also aligns with the broader argument behind functional 3D scene graphs: not every meaningful room relationship is purely spatial, and some structure needs to capture what objects are for and how they support one another~\cite{Zhang_2025_CVPR_OpenFunGraph}. That scene program functions as the semantic contract for the rest of the backend.

This stage became necessary because prompt text by itself was too ambiguous to carry safely through the compiler. Earlier attempts leaned too heavily on prompt parsing and small heuristics, which made the planner brittle and encouraged prompt-specific fixes inside later stages. A prompt such as ``a cozy reading room with rain outside the windows'' mixes mood, objects, and scene cues together. If those signals stay as raw text, later parts of the pipeline have to guess what matters.

The scene-program representation fixes that by normalizing the model output before it is accepted. Unsupported runtime-role collapse is delayed, malformed relations are rejected, and valid archetype labels are trusted only when they match the project's fixed ontology. The contract also now carries explicit design-brief fields, semantic slots, ``do not add'' and negative-constraint structure, and grounded-slot bookkeeping after asset selection, which reduced the need for older scene-token exclusion and role-injection heuristics. This follows the same general lesson seen in LayoutGPT and Holodeck: the model should describe what the room should contain and how the parts relate, while later layers decide whether the room can actually be built~\cite{Feng_2023_NeurIPS_LayoutGPT, Yang_2024_CVPR}. In practice, this let the backend drop several earlier heuristics, including hand-written role injection for otherwise valid model outputs.


\subsubsection{Approved Asset Pipeline}
The approved asset pipeline is the reviewed inventory that the planner and runtime are allowed to use. Prefabs are scanned, enriched with Unity-side metrics, and then placed in review queues where assets can be approved, excluded, or skipped. Semantic overrides are stored separately so that an approved asset is not automatically trusted as the right answer for every role. After review, the backend builds a planner-safe pool and a runtime realization registry from the approved set only, following the same general asset-constrained direction seen in ProcTHOR and Holodeck~\cite{Deitke_2022_NeurIPS, Yang_2024_CVPR}. For the current MVP, that pool is also content-pruned: FoodPack and food-like tabletop clutter are removed from the active planner, runtime, review, material-review, and thumbnail surfaces so that weak decor does not enter generation as a selection problem.

This pipeline became necessary because the raw Unity project was too noisy to use directly. Asset names often contain misleading tokens, and some prefabs that look semantically useful in text are poor matches in practice. A prefab called ``DeskPlant'' may not be an appropriate desk, and a ``TableNumber'' prop should not satisfy a required table role just because the word ``table'' appears in its name. In a Quest-focused VR project, mistakes of that kind become visible quickly because scale, collision, and material problems all reach the user directly.

The approved asset pipeline addresses this by separating two different questions: whether an asset is allowed in the system, and what role that asset is allowed to satisfy. The planner therefore searches a bounded inventory whose assets have already passed at least a minimal semantic and performance check, rather than searching every prefab in the repository. In practice, some categories were removed entirely for the MVP when they repeatedly reduced demo quality without being necessary to core room generation. That narrower operating space strongly improved selection quality and reduced cases where the room was technically compilable but obviously wrong to a human observer. It also enabled the candidate-selection stage to become thinner and more legible: the backend now focuses primarily on legality, packing, and registry-backed filtering, while candidate preference, rejection, and fallback ordering are pushed into the structured model output.

\subsubsection{Indexing, Metadata Enrichment, and Review}
The indexing stage turned the raw Unity repository into something the planner could actually trust. Prefabs were scanned, thumbnails were captured, and AI-assisted metadata passes extracted candidate labels, affordances, scale hints, front-axis information, and other semantic tags from names, geometry summaries, and preview images. Those model-generated labels were useful for throughput, but they were not treated as authoritative. They existed to accelerate curation, not to replace it.

Human review was still the deciding step. Assets were moved through repeated review passes, corrected when the AI-derived labels were wrong or too broad, and only then approved into the planner-safe inventory. In total, more than 1{,}000 assets were manually reviewed across multiple passes before they were allowed into the approved pool. That review process mattered because weak metadata did not stay confined to a spreadsheet. It directly affected selection quality, placement quality, and runtime plausibility.

Direction-aware seating metadata is a good example. Earlier metadata passes knew that an object was a chair, but not which side of the prefab was visually the front. That meant a room could satisfy the relation graph numerically while still looking wrong because the chair's back faced the table. Adding explicit front-axis metadata during indexing, then carrying it into compilation, made those scenes read much more naturally.

\subsection{Implementation}
The most important design choices are to use an LLM for semantic planning and asset selection while keeping validation, runtime safety, and asset eligibility explicit in the compiler. A validated room-shell compile stage, approved-only asset selection, explicit safe-spawn checks, and a backend-generated runtime realization registry were chosen to preserve reliability and comfort on standalone VR hardware. In practice this means the language model proposes what the room should be and which approved assets should realize it, while the backend decides whether that proposal is acceptable for VR use and solves the final grounded layout deterministically.

Early tests made it clear why spatial awareness mattered. Even when the selected assets were broadly correct, the room could still look wrong if objects were dropped into open space without much regard for one another. Desks, chairs, and other props read less like a furnished room and more like a pile of disconnected objects. That problem pushed the project toward relation-aware placement, stronger scene context, and stricter compile-time checks.

The implementation work therefore moved in two directions at once. The planner became more structured and less ad hoc, while the compiler became stricter and more explicit about what counted as a valid room. Those changes needed each other. A stronger planner without a careful compiler would still allow unsafe or awkward layouts. A strict compiler without a meaningful planner would reject too many interesting prompts. The system only became usable once both sides improved together.

\subsubsection{How Spatial Placement Was Implemented}
Spatial placement is handled by a deterministic solver after semantic planning and asset selection are complete. The backend first builds a room shell, grounds selected semantic slots to approved assets and runtime roles, and converts those assets into simplified geometry profiles. Candidate placements are then sampled deterministically over the room using low-discrepancy Halton sequences before being filtered and scored against the normalized scene program, relation graph, clearance constraints, symmetry preferences, and walkway-preservation hints. This still follows the broad pattern emphasized by LayoutGPT and Holodeck, where semantic structure is made explicit before final world construction~\cite{Feng_2023_NeurIPS_LayoutGPT, Yang_2024_CVPR}. The language model authors the semantic and compositional contract, but the final grounded arrangement is chosen by the deterministic placement stack rather than by raw coordinate prediction.

The richer scene program gives that placement stage more direction than a flat role list did. It specifies semantic slots, support hierarchy, relation graph, density target, walkway-preservation hints, and optional-placement intent, so the solving request is much closer to room composition than to a simple list of nouns. GraphDreamer and CommonScenes both argue that explicit relational structure improves scene coherence~\cite{Gao_2024_CVPR_GraphDreamer, Zhai_2023_NeurIPS_CommonScenes}, and that same idea is used here to give the deterministic solver more meaningful spatial work. The compiler still checks the result afterward for safety and runtime suitability, but the arrangement itself now comes from a validated deterministic search over grounded candidates.

\subsubsection{Safe Spawn and Portal Readiness}
One of the simplest but most important implementation details is the safe-spawn search. After the room shell and placements are compiled, the backend searches for a valid player spawn by testing candidate points on the walkable floor while avoiding occupied space. The portal is not allowed to open unless this step succeeds. That means a room can be semantically valid and still be rejected if it does not contain a safe place to enter.

This gate is part of what makes the architecture feel more like a compiler than a demo script. The system does not assume that a plausible room is good enough. It proves that the player can enter it safely, that the floor is valid for teleport locomotion, and that the backend can produce a complete runtime contract before the Unity client commits to the transition.


The safe-spawn search itself is simple but important. It uses a deterministic concentric-ring search over walkable floor candidates while rejecting occupied or unsafe regions, instead of assuming the center of the room will always be usable.

This check matters for both comfort and credibility. From a comfort standpoint, the user should never emerge from the portal inside a chair, too close to a wall, or facing a cluttered corner with no space to move. From a systems standpoint, safe spawn is the final proof that the room is not only a semantic answer to the prompt but also a usable destination. In that sense, spawn selection functions as a practical end condition for the compiler: if a safe spawn cannot be found, the room is not ready.

\subsubsection{Runtime Realization and Decor}
The runtime does not simply instantiate a list of arbitrary prefabs. It consumes a contract that includes exact asset IDs where possible, slot-grounded role behavior where necessary, and scene context that helps the Unity side present the room more coherently. The backend also emits a decor plan and room archetype so that rooms can receive supporting props and simple wall treatment without the headset inventing its own interpretation. Over time, more optional-decor and placement preference logic was moved out of Unity-side heuristics and into this backend contract, which made the manifest a clearer source of truth. Decor is now treated as a gated extension of an already functional room rather than as a cosmetic patch for incomplete layouts.

This runtime contract became richer as the backend design settled. Earlier versions sent placements with less context, which left more room for Unity-side guesswork. That led to inconsistent realizations and made debugging harder because some room logic existed only on the headset. Once more of that intent moved into the manifest, including grounded slots, archetype, decor guidance, exact realization information, and richer scene context such as concept labels and style descriptors, the backend became the clear source of truth. Unity could then focus on realizing the checked room instead of deciding what the room should mean.

\subsection{Research Questions and Goals}
My research question is whether a prompt-driven system can convert natural language into a walkable VR world quickly enough to remain interactive while still enforcing safety, validity, and runtime constraints. The main goals are to generate a validated WorldSpec from a prompt, compile it into a room the user can safely enter, and measure whether this pipeline remains reliable on standalone VR hardware.

\subsection{Data, Tools, and Systems}
The system uses manually reviewed asset packs, stylekit manifests, WorldSpec and manifest JSON contracts, and compiled phase0 artifacts as its primary data sources. The implementation uses Unity with C\# and OpenXR on the client side, and Python modules for planning, validation, substitution, safe-spawn search, decor planning, runtime realization, and compilation on the backend, with language models handling design-brief generation, semantic planning, and approved-asset selection. On the backend, geometry-aware parts of the pipeline rely on standard scientific and spatial libraries such as NumPy, SciPy, and Shapely for profiling, sampling, auditing, and validation. Backend-generated scene context drives wall color, supporting decor, and exact or role-based runtime realization, which reduces the amount of ad hoc logic left on the headset.

\section{Results and Findings}
Evaluation focused on two simple questions: could the backend keep producing valid runtime contracts, and did the rooms start to feel more like places than piles of props? To answer that, the project relied on three forms of checking. Schema and contract tests verified that WorldSpec, manifest, and phase0 artifacts stayed valid. End-to-end backend runs exercised the live planning and placement path. Visual inspection in Unity caught the failures that did not show up cleanly in JSON, such as floating props, empty-feeling rooms, or layouts that were technically legal but still looked wrong. That framing is close in spirit to SceneEval, which argues that text-conditioned indoor scenes should be judged not only on explicit prompt match but also on implicit plausibility conditions such as support, collisions, and navigability~\cite{Tam_2026_WACV_SceneEval}.

What showed up again and again was fairly simple. Early rooms were often valid enough to load, but they did not reliably read as intentional spaces. A desk, chair, and lamp might all be present, yet the result still felt scattered. Once the scene program carried clearer relationships, the approved asset metadata was cleaned up, and the backend started passing stronger scene context into runtime realization, the rooms became easier to read as studies, lounges, bedrooms, or workshops.

The most important result is not unlimited generative range. It is the fact that the backend can reject bad plans, compile the viable ones, and hand Unity a contract it can execute without guessing at the room's meaning on-device. That separation between planning and runtime execution ended up being one of the most useful design decisions in the project.

Backend evaluation was still valuable even when the headset was out of the loop because it separated different kinds of failure. Some prompts failed at the semantic stage. Others failed because the reviewed asset pool could not satisfy a required slot or could not ground a semantic request to an approved asset. In other cases, the deterministic layout search could not produce a safe arrangement or left no usable spawn after furnishing. Those distinctions mattered. They made it possible to improve the right part of the pipeline instead of treating every miss as a vague generation problem.

The strongest improvements did not come from model choice alone. They came from the surrounding structure around it. Once the planner carried clearer relations, stronger negative constraints, and richer scene context, and once the reviewed asset pool stopped surfacing obviously wrong candidates, rooms became easier to read as intentional spaces rather than legal but awkward compilations.

There were limits as well. Rooms anchored by one strong object, such as a bed or desk, improved faster than rooms whose identity depended on subtle staging. Layered decor, shared-line arrangements, and atmosphere were harder to maintain than simple role coverage. That gap points to one of the clearest lessons from the work: matching the nouns in a prompt is not the same as making the room feel right.

\section{Discussion}
One thing became clear early in the project. The LLM was much more useful when it was given a structured planning job than when it was left to improvise from raw prompt text. Asking first for a design brief and then for semantic slots, support hierarchy, relation graph, density, walkway intent, and style cues produced far better behavior than asking for a room in one unconstrained shot. That lines up with the direction suggested by LayoutGPT and Holodeck, and it also held up in practice here~\cite{Feng_2023_NeurIPS_LayoutGPT, Yang_2024_CVPR}.

The checking stage exposed the other side of that tradeoff. It made the system easier to reason about, but it also made weak priors impossible to hide. Thin asset coverage, weak decor policies, and bad semantic metadata showed up immediately once the compiler started checking placements instead of simply accepting them. Spatial placement made this especially clear. A room can look plausible at first glance and still have overlaps, cramped clearances, or weak grouping. The graph-first ideas in GraphDreamer and CommonScenes improved composition, but they did not remove the need for explicit VR-facing checks afterward~\cite{Gao_2024_CVPR_GraphDreamer, Zhai_2023_NeurIPS_CommonScenes}.

The approved asset pipeline turned out to matter more than expected. Manual review is expensive, but the project improved once approved assets, semantic overrides, runtime realization mapping, and decor policies were treated as core data rather than helper files around the planner. In a VR project, asset quality and semantic correctness are not cosmetic issues. They change whether the room feels believable and whether the user trusts it.

Taken together, the project supports a staged approach rather than a single magic generation step. The language model does valuable work when it carries a large share of the semantic and spatial burden. The compiler still has to handle legality, walkability, and runtime predictability. That split does not solve every composition problem, but it does make the system easier to test, explain, and improve.



\section{Conclusion}
This project focused on indoor, room-scale environments built from a fixed and manually reviewed asset library. That scope is narrow, but it made the core question manageable: can a prompt become a room that is safe, walkable, and believable enough to enter in VR? The answer is yes, provided the system treats world generation as a staged process rather than as a single act of generation.

The strongest limitations are also clear. Asset coverage still matters more than any prompt trick. If the reviewed inventory is thin, the planner and realization registry run out of good options quickly. The rooms also become less convincing once they depend on subtle staging instead of a few strong anchor objects. Multi-room layouts, outdoor scenes, and richer atmosphere would require a broader world representation than the room-shell compiler used here.

The next directions for future work are fairly clear. Expanding the reviewed inventory would improve prompt coverage immediately. A broader WorldSpec could support multi-room and outdoor environments. Richer staging logic could push the system beyond basic furnishing and toward stronger orientation, grouping, and circulation. Geometry synthesis, as explored in work such as Text2Room~\cite{hoellein2023text2room}, remains a longer-term option for prompts that fall outside any curated asset library, but only if the resulting geometry can be checked against the same walkability and performance requirements.

Even with those limits, the project shows that prompt-driven VR world generation can be treated as a practical engineering problem. Structured semantic planning, reviewed assets, explicit validation, relation-aware placement, and runtime contracts were enough to produce rooms that were not just descriptively related to a prompt, but usable as destinations. That is a modest result compared with unrestricted world generation, but it is a meaningful one for live XR systems.
